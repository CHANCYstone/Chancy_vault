**📚 前10分钟：数据仓库建模核心概念速通**

  

**✅ 一、数据仓库的“分层架构”模型（经典五层）**

|**层级**|**名称**|**作用**|
|---|---|---|
|ODS（操作数据层）|原始日志|不做业务加工，只做清洗|
|DWD（明细数据层）|精细结构化|以“事件”或“事实”为粒度，一行一件事|
|DWS（汇总数据层）|业务宽表|聚合后的主题表（如用户支付总览）|
|ADS（应用数据层）|指标报表|直接对接分析、可视化层|
|DIM（维度层）|维度表|提供枚举/属性支持，如“国家、支付方式”|

🧠 面试时你可以说：

  

> “我熟悉分层建模方法，实际建模时会通过 ODS → DWD → DWS → ADS 的路径逐层提纯数据，既提升可维护性，也方便后续指标复用。”

---

**✅ 二、事实表 VS 维度表**

• **事实表**：记录“发生了什么”——如支付流水、订单行为，一般会有时间戳、金额、用户ID

• **维度表**：描述“对象的属性”——如用户表（性别、地区）、支付方式表（名字、类型）

  

🧠 面试答法：

  

> “我们遵循星型模型，主表是订单事实表，维表如用户维度表、支付渠道维度表通过外键进行维度丰富。”

**🛠️ 第一步：识别业务实体**

  

我们需要建模的主要表：

• 订单表（order_fact）【事实】

• 用户表（user_dim）【维度】

• 支付方式表（payment_channel_dim）【维度】

• 支付流水表（payment_fact）【事实】

• 退款表（refund_fact）【事实】

**🧩 第二步：表设计示例（简化字段）**

  

**order_fact**

|**字段名**|**类型**|**说明**|
|---|---|---|
|order_id|string|订单唯一ID|
|user_id|string|用户ID|
|create_time|timestamp|下单时间|
|status|string|状态（已支付/未支付）|
|total_amount|decimal|下单金额|

**payment_fact**

|**字段名**|**类型**|**说明**|
|---|---|---|
|payment_id|string|支付流水ID|
|order_id|string|关联订单|
|pay_time|timestamp|支付时间|
|amount|decimal|支付金额|
|channel_id|string|支付渠道|

**user_dim**

|**字段名**|**类型**|**说明**|
|---|---|---|
|user_id|string|用户ID|
|country|string|国家|
|register_time|timestamp|注册时间|

**payment_channel_dim**

|**字段名**|**类型**|**说明**|
|---|---|---|
|channel_id|string|渠道ID|
|channel_name|string|支付方式名称（PayPal/信用卡/Apple Pay）|

**🔍 第三步：DWS 层示例建表（用户支付宽表）**

```
-- 用户支付行为汇总宽表
CREATE TABLE dws_user_payment_summary AS
SELECT
  u.user_id,
  u.country,
  COUNT(DISTINCT o.order_id) AS total_orders,
  COUNT(DISTINCT p.payment_id) AS total_payments,
  SUM(p.amount) AS total_payment_amount,
  MIN(p.pay_time) AS first_payment_time
FROM order_fact o
JOIN payment_fact p ON o.order_id = p.order_id
JOIN user_dim u ON o.user_id = u.user_id
WHERE o.status = '已支付'
GROUP BY u.user_id, u.country;
```

这个表就可以支持：

• 用户首付时间

• 每用户的支付总额

• 国家维度的聚合分析
